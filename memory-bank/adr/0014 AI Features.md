# Agent Session: The AI Loop
AI agent activities are grouped into sessions similar to `topics`. Sessions are persisted so that a session can be ended or restarted at a later time.
During each session, a `Loop` continues until it's terminated, performing the following steps of tasks:
1. Reveila sends a "Tool Definition" + "System Instructions" + "User Intent."
2. LLM processes the "Reasoning" and returns the "info" requested or "actions" that need to be performed (structured JSON tool-call instructions).
3. Reveila parses the JSON, execute the too-call (e.g. runs PowerShell script or performs an API call) and captures the result.
5. Reveila sends the result back to the LLM to "close the loop."

# LangChain4j as the "Translation Layer" between Reveila and AI Models

* Abstraction: LangChain4j converts Java code to the specific API spec of the provider (OpenAI vs. Gemini).
* Type Safety: It turns the AI's JSON response back into a Java Object automatically.
* Resilience: If we want to switch from a Cloud Model (OpenAI) to a Local Model (Ollama) because of privacy concerns, we only change one line of code.

# The "Message Chain" Pattern

When the system sends the execution results back to the AI Model, it must send a list of messages that represents the full history of the "thought process".

By using dev.langchain4j.memory.ChatMemory, the system maintains a chronologically ordered list of all interactions within a session. When the tool execution results are sent back to "close the loop," they are added to this chain, and the entire history is provided to the AI model for its next reasoning step.

The Message Chain is implemented in AgenticFabric.java. Its `processLoop()` method explicitly builds this chain, ensuring the AI has a complete "thought process" history for every turn of the loop.

* SystemMessage: Contains the base instructions and MCP tool definitions.
* UserMessage: The initial intent or task provided by the user.
* AiMessage: Captures the LLM's reasoning and the specific tool-call instruction.
* ToolExecutionResultMessage: Contains the output from the tool (e.g., PowerShell result, API response).
* Final AiMessage: The LLM's summary or final answer, generated by seeing the full chain of the previous four steps.

# 📊 Prompt Format

We will use **Markdown + XML** format for prompts (SystemMessage or UserMessage). The Prompt.getBasePrompt() method implements this pattern.

Markdown: Used for high-level structure and readability (e.g., # ROLE, # CONTEXT, ## 🧠 Reasoning).
XML Tags: Used as structural "firewalls" to clearly delineate dynamic content and prevent prompt injection (e.g., <context_boundary>, <objective>).

Format Performance Comparison:

Feature             Plain Text      JSON        Markdown + XML (Recommended)
LLM Reasoning       Basic           Moderate    Highest (81.2% Success Rate)
Token Cost          Lowest          High        Medium (Optimized)
Injection Safety    Poor            Good        Excellent (Tags act as firewalls)
Human Readability   High            Low         High

# Managing "Context Windows"

* Tokens = Money/Memory: Every time you send the "original prompt + history + result," the request gets larger and more expensive (or slower).
* The "Sliding Window": For Reveila-Suite, your Java code should implement a "Summary" or "Truncation" strategy. After 10-15 turns, you might have the AI "summarize" the old parts of the conversation to save space while keeping the "State" intact.
* Java Implementation: With LangChain4j, you don't have to manually manage this array of strings. You use a ChatMemory component.
```java
// How Reveila-Suite keeps track of the "State"
ChatMemory chatMemory = MessageWindowChatMemory.withMaxMessages(20);

// When the user says "Yes" to your prompt...
chatMemory.add(UserMessage.from("Yes, run the script."));

// When the script finishes...
String result = terminal.execute(script);
chatMemory.add(ToolExecutionResultMessage.from(toolId, result));

// The next call to the AI automatically includes EVERYTHING in chatMemory
String finalAnswer = model.generate(chatMemory.messages()).content();
```
## Session History Size Management

Storing Chat Sessions can accumulate memory and storage size quickly. We use the reveila.properties ai.optimization.history=10MB (acceptal values can be MB or unlimited) to limit the session history size to this limit, both in memory and in persisted storage.

Implementation Details:

* Configuration: ai.optimization.history=10MB in reveila.properties defines the maximum total memory footprint allowed for all stored chat sessions.
* Size-Aware Session Management: The estimateSessionSize method in AgentSessionManager calculates the memory footprint of a session based on the length of its ChatMemory messages and context map content.
* The saveSession method now enforces the ai.optimization.history limit. Before persisting a new session, it checks the total size of the store; if the limit is exceeded, it automatically evicts the oldest (Least Recently Used) sessions until the footprint is within the configured boundary.

# The Autonomous Agent (For Personal Edition)

For the Reveila Personal Edition, we will implement autonomous AI agent, that is, allowing the agent to act autonomously if authorized by the user.

Workflow:

Step		    Actor	        Action
1. Intent	    User	        "Find all large log files over 500MB and compress them."
2. Reasoning	LLM	            Realizes it has no pre-made script for this. Decides to write one.
3. Generation	LLM	            Sends back a full PowerShell block using Get-ChildItem and Compress-Archive.
4. Execution	Reveila-Suite	Your Java code receives the raw string, saves it to a temp .ps1, and runs it.
5. Feedback	    Reveila-Suite	Captures the output/errors and sends them back to the LLM to confirm success.

# Pre-Defined Tools Call (For Enterprise Edition)

For Reveila Enterprise Edition, we only allow AI Agent to execute a list of predefined tools that are secure, "lite," and scriptable.

Suggested Solution Path:

Category	    Candidate Solution	    Why?
Terminal	    Xterm.js + node-pty	    Industry standard for web-based/lite terminals; very stable.
Automation	    Playwright	            Better "Agentic" support than Selenium; handles Modern Web easily.
Security	    Wasmtime	            Run agent logic in WebAssembly for near-native speed but total isolation.
Context	        ChromaDB or DuckDB	    Both are "Lite" and can run locally without a heavy server setup.

The Workflow:
The integration follows a "Loop" where the Java application acts as the Orchestrator and the LLM acts as the Reasoning Engine.

Phase	            Actor	            Responsibility
1. The Goal	        User	            Provides a high-level intent: "Check my system health and clear old caches."
2. Prompt Assembly	Java Orchestrator	Gathers "Context" (OS type, available tools, project path) and wraps the User Goal into a System Prompt.
3. The Format	    JSON / Tool Call	Use Function Calling (Tools API) instead of plain English.
4. The API Spec	    Java Orchestrator	Formats prompt into JSON required by the provider (OpenAI, Anthropic, or a local Ollama).
5. The Response	    LLM (Reasoning)	    The model responds with a Tool Call request (e.g., {"tool": "runPowerShell", "args": {"scriptName": "CheckHealth.ps1"}}).
6. The Interpreter	Java Orchestrator	Parses the JSON response, validates security, and executes the actual Java method.

Implementation: ReveilaTerminal.executeSafeScript()

# The Efficiency vs. Security Trade-off

All Tool calls must observe:

1. Execution Policy: Scripts must run with -ExecutionPolicy under a Low-Privilege User Account (Service Account) specifically for Reveila.
2. Timeouts: Hard-cap every script at 30 seconds. This prevents the LLM from accidentally writing an infinite loop that freezes the system.
3. The "Human-in-the-Loop" (HITL) Proxy: For the Personal Edition, the UI should show the script with a [Run] / [Edit] / [Cancel] button before execution.
